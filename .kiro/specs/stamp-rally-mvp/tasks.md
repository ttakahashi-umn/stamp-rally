# Implementation Plan

- [x] 1. Foundation: 参加者別スタンプラリー基盤の準備
- [x] 1.1 認証・端末セッション向け永続化基盤を整備する
  - 参加者と端末セッションを分離し、セッション識別子は平文保存せず照合可能にする
  - Observable: 認証情報保存時に平文セッション識別子が永続化されない
  - _Requirements: 1.2, 2.1_
- [x] 1.2 会場・押印履歴・進捗向け永続化基盤を整備する
  - 会場と押印履歴を参加者単位で管理し、重複押印を防ぐ一意制約を適用する
  - Observable: 同一参加者・同一会場の重複押印がDB制約で拒否される
  - _Requirements: 3.2, 6.2_
- [x] 1.3 認証前提の共通リクエスト検証とエラー応答基盤を整備する
  - 未認証・期限切れ・不正入力を同一方針で扱う
  - Observable: 未認証リクエストが統一フォーマットで拒否される
  - _Requirements: 1.2, 2.3, 5.3_
- [x] 1.4 バックエンドの自動テスト実行基盤を本機能向けに拡張する
  - 認証、押印、進捗の主要シナリオを検証可能にする
  - Observable: バックエンドの新規機能テストをCI相当手順で実行できる
  - _Requirements: 1.2, 3.2, 5.1_
- [x] 1.5 フロントエンドの自動テスト実行基盤を本機能向けに拡張する
  - 起動導線、失敗表示、再試行導線を検証可能にする
  - Observable: フロントエンドの主要導線テストを実行できる
  - _Requirements: 1.1, 5.2, 6.3_

- [x] 2. Core: バックエンド認証・押印ドメインの実装
- [x] 2.1 (P) メール導線トークンの初回認証と端末再訪セッションを実装する
  - 初回認証成功時に再訪可能なセッションを発行する
  - 失効時は再認証へ誘導できる判定結果を返す
  - Observable: 初回認証APIが有効トークンで成功し、無効トークンで拒否する
  - _Requirements: 1.1, 1.2, 2.1, 2.2_
  - _Boundary: Auth API, Session Service_
- [x] 2.2 (P) 会場参照と参加者別進捗集約のリポジトリ/サービスを実装する
  - 参加者単位で取得済み会場数と一覧状態を返せるようにする
  - Observable: 進捗APIが参加者単位の total/completed/items を返す
  - _Requirements: 6.1, 6.2, 6.3_
  - _Boundary: Venue Repository, Stamp Record Repository, Progress Service_
- [x] 2.3 署名付きQRの真正性検証とTTL検証を実装する
  - 無効署名・期限切れQRを押印前に拒否する
  - Observable: 無効署名/期限切れQRで押印処理が停止し理由が返る
  - _Depends: 2.1_
  - _Requirements: 3.1, 5.1_
- [x] 2.4 位置判定・重複防止を含む押印判定を実装する
  - 位置情報は押印判定用途に限定し、不要な永続化を行わない
  - 位置情報欠落/範囲外/重複の失敗理由を区別して返す
  - Observable: 押印APIが成功/重複/範囲外/位置情報不足を判別して応答する
  - _Depends: 2.2, 2.3_
  - _Requirements: 3.2, 4.1, 4.2, 4.3, 5.3_
- [x] 2.5 押印APIと進捗APIの公開契約を最終化し、再試行可能なエラー分類を提供する
  - 通信失敗時に成功未確定として扱える応答カテゴリを定義する
  - Observable: クライアントが再試行可否を判別できるエラー応答を受け取れる
  - _Depends: 2.4_
  - _Requirements: 3.3, 5.2, 6.2_

- [x] 3. Core: 参加者向けフロントエンド体験の実装
- [x] 3.1 (P) 認証ゲートと再訪判定フローを実装する
  - メール導線起動時に認証済み/未認証/失効を分岐表示する
  - Observable: 初回起動と再訪起動で画面遷移が要件どおりに変化する
  - _Depends: 2.1_
  - _Requirements: 1.1, 1.3, 2.1, 2.2_
  - _Boundary: AuthGate_
- [x] 3.2 (P) スタンプ一覧と達成状況表示を参加者別進捗モデルへ置き換える
  - 一覧取得失敗時に理由表示と再読み込み操作を提供する
  - Observable: 一覧取得失敗時に再読み込み操作で復帰できる
  - _Depends: 2.2_
  - _Requirements: 6.1, 6.2, 6.3_
  - _Boundary: StampList_
- [x] 3.3 (P) 押印実行UIにQR入力と位置情報許可フローを実装する
  - 位置情報拒否時に押印不能メッセージと再試行導線を表示する
  - Observable: 位置情報拒否時は押印できず、許可時のみ押印要求が送信される
  - _Depends: 2.5_
  - _Requirements: 3.1, 4.2, 4.3, 5.1_
  - _Boundary: StampScanAction_
- [x] 3.4 押印結果の失敗理由表示と再試行導線を統合し、押印後進捗を即時再取得する
  - 押印成功/失敗の結果表示と一覧更新を同一画面で完結させる
  - Observable: 通信エラー時に成功未確定表示と再試行操作が可能になる
  - _Depends: 3.2, 3.3_
  - _Requirements: 3.3, 5.2, 5.3, 6.2_

- [x] 4. Integration: 境界横断の結合と運用前提の確立
- [x] 4.1 バックエンドとフロントエンドの契約整合を検証し、失敗理由マッピングを統一する
  - 押印失敗理由がAPIとUIで一貫した文言カテゴリになるよう調整する
  - Observable: 代表的な失敗ケースでAPIコードとUI表示が期待どおり一致する
  - _Depends: 2.5, 3.4_
  - _Requirements: 5.1, 5.2, 5.3_
- [x] 4.2 端末セッション失効と再認証導線の統合シナリオを完成させる
  - 失効後に押印操作へ進んだ際、再認証へ安全に戻せるようにする
  - Observable: 失効セッションで押印操作時に再認証フローへ遷移する
  - _Depends: 2.1, 3.1, 3.3_
  - _Requirements: 1.2, 2.2, 2.3_
- [x] 4.3 押印イベントの観測指標（成功率・拒否理由・位置情報拒否率）を記録できるよう統合する
  - 運営が失敗傾向を追跡できる最小限の計測イベントを揃える
  - Observable: 押印操作で成功/失敗種別の記録が出力される
  - _Depends: 2.4, 2.5_
  - _Requirements: 5.3_

- [x] 5. Validation: 受け入れ基準検証
- [x] 5.1 バックエンド単体テストで認証・押印判定・進捗集約を検証する
  - 認証失敗、期限切れQR、位置情報不一致、重複押印を網羅する
  - Observable: テスト実行で主要判定ロジックがすべて成功する
  - _Depends: 2.5_
  - _Requirements: 1.2, 3.2, 4.2, 5.1, 6.2_
- [x] 5.2 API統合テストで認証→押印→進捗更新の一貫性を検証する
  - 再試行可能エラーと再認証導線をAPIレベルで検証する
  - Observable: 統合テストで主要APIシナリオが成功する
  - _Depends: 4.1, 4.2_
  - _Requirements: 1.1, 2.3, 3.3, 5.2, 6.2_
- [x] 5.3 フロントエンド統合テストで起動→認証→押印→進捗更新を検証する
  - 一覧取得失敗時の再読み込み復帰を検証する
  - Observable: UI統合テストで主要導線と失敗復帰シナリオが成功する
  - _Depends: 3.4, 4.1_
  - _Requirements: 1.1, 1.3, 5.2, 6.3_
- [x] 5.4 同時押印を想定した負荷スモーク検証を実施する
  - 競合時の失敗率と応答時間を測定し、運用リスクを可視化する
  - Observable: 同時押印時の成功率と応答時間がレポート可能な形で取得される
  - _Depends: 2.5, 4.3_
  - _Requirements: 3.1, 5.3_
- [x]* 5.5 追加のUIレンダリング回帰テストを補強する
  - MVP受け入れを阻害しない範囲で画面表示崩れ検知を追加する
  - Observable: 回帰テスト群で主要表示コンポーネントの崩れを検知できる
